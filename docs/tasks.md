## 6. Parallelization Rules

### 6.1 Suitable For Parallel Work

以下任务适合并行，前提是文档规则已经冻结：

- 首页骨架结构
- 历史记录页骨架结构
- 仪表盘页骨架结构
- “我”页骨架结构
- 底部导航的静态结构
- 页面文案与空状态占位补齐

原因：

- 四个一级页面结构清晰，耦合点主要在底部导航和共享状态入口
- 在不改产品规格和数据模型的前提下，页面骨架可以分头推进

### 6.2 Must Not Run In Parallel

以下任务不应并行推进，避免规则互相踩踏：

- 修改 `docs/product-spec.md` 与实现首页启动逻辑
- 修改 `docs/data-model.md` 与实现状态管理
- 多人同时改同一个页面文档
- 多人同时改 `docs/tasks.md` 任务优先级
- 在 Terminal D 未收口前，同时开始多条实现线的交互联调

原因：

- 这些任务对规则源头、依赖顺序和合并冲突敏感
- 一旦边界没收紧，实现很容易把错误写死

### 6.3 Sequential Dependencies

建议依赖顺序：

1. 产品规格、数据模型、信息架构三份文档收口
2. 四个一级页面静态骨架并行推进
3. 本地状态管理接入
4. 首页与历史记录的关键交互接线
5. 仪表盘统计接线
6. “我”页管理动作接线
7. 联调与规则回归

## 7. Terminal Split

### Terminal D

职责：

- 作为总负责人角色，直接和用户协作
- 负责目标澄清、范围控制、任务拆分、优先级排序
- 协调 Terminal A / B / C 的工作边界
- 在实现开始前做统一收口，避免各线偏航

重点文件：

- `README.md`
- `docs/product-spec.md`
- `docs/data-model.md`
- `docs/information-architecture.md`
- `docs/tasks.md`
- `docs/multi-terminal-workflow.md`
- `docs/terminal-d-lead.md`

说明：

- 默认由主 Codex 会话承担，不必额外单开

### Terminal A

职责：

- 维护产品规则
- 校验实现是否符合产品规格

重点文件：

- `docs/product-spec.md`
- `README.md`

### Terminal B

职责：

- 维护数据模型
- 约束本地存储结构和状态流转

重点文件：

- `docs/data-model.md`

### Terminal C

职责：

- 维护页面信息架构
- 输出页面骨架与任务执行清单
- 标记哪些任务可并行，哪些需等待上游收口

重点文件：

- `docs/information-architecture.md`
- `docs/tasks.md`

## 8. Collaboration Rule

- 主 Codex 会话默认承担 Terminal D 职责，A / B / C 不直接替代 D 做总控决策
- 先更新对应文档，再开始实现
- 发生冲突时，以已落盘文档为准，而不是以聊天记忆为准
- 对齐阶段默认一次只问一个问题
- 页面骨架可以并行，但状态管理和规则改动必须先经过 D 收口
- Terminal C 不改数据模型，不仲裁产品规则，只记录结构风险并回抛给 D

## 9. Risks
- 仪表盘统计卡片在固定口径和周期切换之间还需要一次文案统一，否则实现阶段容易各写各的。
